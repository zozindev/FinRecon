# Portfolio Summary

## 한 줄 소개

FinRecon은 외부 결제 거래 CSV와 내부 원장을 검증, 대사, 정산하는 Java/Spring Boot 백엔드 프로젝트입니다.

## 왜 만든 프로젝트인가

기존 데이터 검증/업무 자동화 경험을 금융 백엔드 도메인으로 확장하기 위해 만들었습니다. 단순 CRUD 대신 실제 운영 업무에 가까운 흐름인 파일 업로드, 행 단위 검증, 거래 대사, 배치 정산, 권한 제어, 감사 추적을 하나의 백엔드 시스템으로 구현했습니다.

## 핵심 시나리오

1. 운영자가 로그인해 JWT를 발급받습니다.
2. 외부 결제 거래 CSV를 업로드합니다.
3. 시스템은 정상 행과 오류 행을 분리 저장합니다.
4. 내부 원장 CSV를 업로드합니다.
5. Spring Batch Job으로 거래와 원장을 대사합니다.
6. `MATCHED` 거래만 기준으로 일일 정산을 실행합니다.
7. 관리자는 배치 이력과 감사 로그를 조회합니다.

## 보여줄 수 있는 역량

- Spring Boot 계층형 백엔드 설계
- JPA/Flyway 기반 명시적 데이터 모델링
- CSV 파일 처리와 행 단위 오류 분리
- 금융성 금액 처리를 위한 `BigDecimal` 사용
- Spring Batch 기반 재실행 가능한 업무 처리
- Spring Security + JWT 기반 인증/권한 제어
- 감사 로그와 배치 실행 이력으로 운영 추적성 확보
- MockMvc 통합 테스트와 GitHub Actions CI
- README, API 명세, ERD, 아키텍처 문서화

## 면접에서 설명하기 좋은 설계 선택

| 질문 | 답변 포인트 |
| --- | --- |
| 왜 오류 행을 따로 저장했나? | 전체 파일을 실패시키면 정상 거래 처리까지 막히므로, 정상 행은 저장하고 오류 행만 운영자가 수정할 수 있게 분리했습니다. |
| 왜 `BigDecimal`을 썼나? | 금융 금액은 부동소수점 오차를 허용하면 안 되므로 Java `BigDecimal`과 DB `decimal(19,2)`를 사용했습니다. |
| 왜 Spring Batch를 썼나? | 대사와 정산은 일자 기준으로 반복 실행되는 업무이므로 Job/Step 구조, 실행 이력, 재시도 흐름을 표현하기 적합합니다. |
| 중복 처리는 어떻게 했나? | 파일은 SHA-256 해시로 중복 업로드를 차단하고, 거래/원장 ID는 DB unique 제약과 서비스 검증으로 막습니다. |
| 감사 로그는 왜 필요한가? | 금융성 운영 업무에서는 누가 언제 업로드, 대사, 정산을 실행했는지 추적 가능해야 하기 때문입니다. |
