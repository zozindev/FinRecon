# FinRecon | Codex 개발용 프로젝트 브리프

## 1. 프로젝트 요청 배경

나는 현재 칸타코리아 Operation 부서 Script팀에서 온라인 설문 프로그래밍과 운영 자동화 업무를 수행하고 있다.

- `HTML`, `CSS`, `JavaScript` 기반 설문 로직 구현
- `Excel VBA` 기반 파일럿 데이터 검증 자동화 도구 개발
- `Python` 기반 스케줄링/업무 자동화 프로그램 개발
- 데이터 정합성 검증, 반복 업무 표준화, 휴먼 에러 감소 경험

앞으로 금융 IT 분야의 개발자로 이직하기 위해 백엔드 포트폴리오를 만들고자 한다. 목표 직무 우선순위는 다음과 같다.

1. `Java/Spring` 기반 금융 백엔드 개발
2. 금융 데이터 처리/배치 개발
3. 업무 자동화 및 내부 시스템 개발

이 프로젝트는 기존의 데이터 검증 및 업무 자동화 경험을 금융 거래 데이터 처리, 정산, 대사, 배치 시스템 개발 역량으로 확장하기 위한 개인 포트폴리오 프로젝트이다.

## 2. 프로젝트 개요

### 프로젝트명

**FinRecon - 금융 거래 검증·정산·대사 플랫폼**

### 한 줄 소개

외부 거래 파일을 업로드하여 유효성을 검증하고, 내부 원장 데이터와 자동 대사한 뒤 일별 정산 결과와 오류 내역을 관리하는 금융 운영 백엔드 시스템

### 프로젝트 목표

- `Java`와 `Spring Boot` 기반의 REST API 서버 개발 역량을 확보한다.
- 금융 데이터의 정합성, 검증, 대사, 정산 처리 흐름을 직접 구현한다.
- `Spring Batch`를 활용하여 파일 처리 및 일마감 배치 개발 경험을 만든다.
- 트랜잭션, 중복 방지, 예외 처리, 감사 로그 등 금융 백엔드에서 중요한 주제를 다룬다.
- 테스트, Docker, CI, 문서화까지 포함한 제출 가능한 GitHub 포트폴리오를 완성한다.

### 중요 원칙

- 실제 금융회사, 실제 고객, 실제 계좌 또는 회사 업무 데이터는 사용하지 않는다.
- 모든 데이터와 업무 규칙은 포트폴리오용으로 설계한 가상의 거래/정산 모델을 사용한다.
- 첫 목표는 완성 가능한 단일 백엔드 애플리케이션이다. 마이크로서비스나 과한 인프라 구성은 초기 범위에서 제외한다.

## 5. 가상 비즈니스 시나리오

FinRecon은 가상의 결제 서비스 운영팀이 사용하는 내부 시스템이다.

매일 외부 결제 파트너로부터 거래 데이터 파일이 들어오며, 내부 시스템에는 거래에 대응하는 원장 데이터가 저장되어 있다. 운영팀은 두 데이터가 일치하는지 확인하고, 정상 거래를 기준으로 일별 정산 금액을 산출해야 한다.

### 업무 흐름

1. 운영자가 외부 거래 CSV 파일을 업로드한다.
2. 시스템은 파일 형식과 개별 거래 값의 유효성을 검증한다.
3. 정상 거래는 저장하고, 오류 거래는 오류 사유와 함께 별도로 기록한다.
4. 내부 원장 데이터와 외부 거래 데이터를 거래번호 기준으로 대사한다.
5. 일별 정산 배치가 정상 대사 완료 거래의 총액, 수수료, 지급 예정 금액을 계산한다.
6. 운영자는 처리 결과, 오류 건, 불일치 건, 정산 결과, 배치 실행 이력을 조회한다.
7. 오류를 수정한 파일은 재업로드하거나 오류 건을 재처리할 수 있다.

## 6. 핵심 도메인 용어

| 용어 | 의미 |
|------|------|
| 외부 거래 (`Transaction`) | 외부 결제 파트너가 전달한 승인/취소 거래 내역 |
| 내부 원장 (`LedgerEntry`) | 서비스 내부 기준으로 기록된 거래 금액 및 상태 |
| 검증 (`Validation`) | 필수값, 값 범위, 형식, 상태 관계, 중복 여부 확인 |
| 대사 (`Reconciliation`) | 외부 거래와 내부 원장이 동일한 거래를 나타내며 금액/상태가 일치하는지 비교 |
| 정산 (`Settlement`) | 정상 대사 거래를 기준으로 일별 지급 금액을 계산하는 처리 |
| 일마감 (`Daily Closing`) | 특정 영업일의 데이터 반영을 마치고 정산 결과를 확정하는 작업 |
| 감사 로그 (`Audit Log`) | 사용자의 업로드, 재처리, 배치 실행 등 주요 행위 기록 |

## 7. 주요 업무 규칙

초기 구현에서는 아래 규칙을 기본 도메인 정책으로 사용한다. 구현 과정에서 변경이 필요하면 문서에 이유를 기록한다.

### 거래 규칙

- 모든 외부 거래는 고유한 `transactionId`를 가진다.
- 동일 `transactionId`의 중복 저장은 허용하지 않는다.
- 거래 유형은 `APPROVAL` 또는 `CANCEL`이다.
- 승인 거래 금액은 `0`보다 커야 한다.
- 취소 거래는 원승인 거래 ID인 `originalTransactionId`를 반드시 가져야 한다.
- 취소 금액은 원승인 거래의 잔여 취소 가능 금액을 초과할 수 없다.
- 거래 상태는 `RECEIVED`, `VALID`, `INVALID`, `RECONCILED`, `MISMATCH`, `SETTLED` 등을 사용할 수 있다.

### 파일 업로드 규칙

- CSV 파일만 허용한다.
- 동일 파일의 중복 업로드를 방지하기 위해 파일 해시값을 저장한다.
- 한 파일에 정상 데이터와 오류 데이터가 섞여 있을 경우 정상 건은 저장하고 오류 건은 오류 내역으로 저장한다.
- 업로드 처리 결과로 전체 건수, 정상 건수, 오류 건수, 중복 건수를 제공한다.

### 대사 규칙

- 외부 거래의 `transactionId`와 내부 원장의 참조 거래번호를 기준으로 매칭한다.
- 거래가 양쪽에 존재하고 금액 및 상태가 일치하면 `MATCHED`이다.
- 외부 거래만 존재하면 `LEDGER_MISSING`이다.
- 내부 원장만 존재하면 `TRANSACTION_MISSING`이다.
- 양쪽에 존재하지만 금액이 다르면 `AMOUNT_MISMATCH`이다.
- 상태가 다르면 `STATUS_MISMATCH`이다.

### 정산 규칙

- 정상 대사된 승인 거래에서 정상 대사된 취소 거래를 차감하여 정산 대상 금액을 계산한다.
- 초기 수수료율은 단순화를 위해 `2.5%` 고정값으로 설정한다.
- 정산 금액은 `정산 대상 금액 - 수수료`로 계산한다.
- 금액은 `BigDecimal`을 사용하고, 소수점 처리 정책을 명시한다.
- 정산 결과는 대상 영업일별로 한 번만 확정하며, 재실행 시 중복 생성되지 않아야 한다.

### 보안 및 운영 규칙

- 고객 식별용 계좌번호 또는 카드번호에 해당하는 필드는 저장하거나 응답할 경우 마스킹한다.
- 사용자 업로드, 대사 실행, 정산 실행, 재처리 작업은 감사 로그에 남긴다.
- 실패한 배치의 원인과 실행 상태를 조회할 수 있어야 한다.

## 9. 예상 데이터 모델

정확한 ERD는 구현 전 Codex가 다시 제안하고 확정한다. 초기 엔티티 후보는 아래와 같다.

| Entity / Table | 주요 책임 |
|----------------|-----------|
| `users` | 사용자, 역할, 인증 정보 |
| `uploaded_files` | 파일명, 해시, 업로드 일시, 처리 상태, 처리 통계 |
| `transactions` | 외부 거래 ID, 유형, 금액, 거래일, 상태, 원승인 참조 |
| `ledger_entries` | 내부 원장 참조번호, 금액, 상태, 기록일 |
| `validation_errors` | 파일/행/필드별 검증 오류와 오류 코드 |
| `reconciliation_results` | 거래-원장 대사 결과, 불일치 유형, 실행 일시 |
| `settlements` | 정산일, 총 승인액, 총 취소액, 수수료, 지급액, 상태 |
| `batch_executions` | 작업명, 시작/종료 시간, 상태, 처리 건수, 오류 메시지 |
| `audit_logs` | 행위자, 행위 유형, 대상, 실행 시간, 메타데이터 |

### 구현 시 고려할 것

- 엔티티와 API 응답 DTO를 분리한다.
- 상태값은 Enum으로 관리한다.
- 금액 타입은 `BigDecimal`을 사용한다.
- 날짜/시간 정책을 정한다. 초기에는 `LocalDate`와 `LocalDateTime`을 사용하고 타임존 정책을 README에 기록한다.
- 중복 방지와 조회 성능을 위한 Unique Constraint 및 Index를 설계한다.

## 10. 예시 CSV 형식

외부 거래 업로드의 초기 입력 형식은 다음으로 시작한다.

```csv
transactionId,transactionType,originalTransactionId,merchantId,transactionDate,amount,maskedPaymentNumber,status
TXN-20260601-0001,APPROVAL,,MERCHANT-001,2026-06-01,12000,****-****-****-1234,APPROVED
TXN-20260601-0002,CANCEL,TXN-20260601-0001,MERCHANT-001,2026-06-01,2000,****-****-****-1234,CANCELED
```

내부 원장 초기 입력 또는 테스트 데이터 형식은 다음으로 시작한다.

```csv
ledgerReferenceId,transactionId,merchantId,recordDate,amount,status
LEDGER-0001,TXN-20260601-0001,MERCHANT-001,2026-06-01,12000,APPROVED
LEDGER-0002,TXN-20260601-0002,MERCHANT-001,2026-06-01,2000,CANCELED
```

## 11. API 범위 초안

API URL과 세부 DTO는 구현 전 Codex가 명세 문서로 정리한 뒤 확정한다.

### Phase 1 대상 API

| Method | Endpoint | 설명 |
|--------|----------|------|
| `POST` | `/api/v1/transaction-files` | 외부 거래 CSV 업로드 및 검증/저장 |
| `GET` | `/api/v1/transaction-files/{fileId}` | 파일 처리 결과 조회 |
| `GET` | `/api/v1/transactions` | 거래 목록 조건 조회 |
| `GET` | `/api/v1/transactions/{transactionId}` | 개별 거래 상세 조회 |
| `GET` | `/api/v1/validation-errors` | 파일 또는 거래별 검증 오류 조회 |

## 13. 권장 패키지 구조 초안

구현 전 프로젝트 스타일에 따라 조정할 수 있지만, 도메인 기준 분리를 우선 검토한다.

```text
src/main/java/.../finrecon/
  common/
    exception/
    response/
    config/
  auth/
  transaction/
    controller/
    service/
    domain/
    repository/
    dto/
  upload/
  ledger/
  reconciliation/
  settlement/
  batch/
  audit/
```

### 구조 원칙

- Controller는 요청/응답 변환과 검증에 집중한다.
- 핵심 업무 규칙은 Service 또는 Domain 계층에 둔다.
- Repository 직접 호출이 Controller에 노출되지 않도록 한다.
- DTO와 Entity를 구분한다.
- 공통 예외 응답 형식을 일관되게 유지한다.

- 승인 거래만 있는 일자의 정산

- 정산 대상이 아닌 불일치 거래 제외

- 권한 없는 사용자의 정산 실행 차단
